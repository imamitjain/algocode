📜 Arrays (Primitive & Object)

Arrays are fixed-size, contiguous memory blocks. Not part of the Collections framework but the foundation for ArrayList, ArrayDeque, PriorityQueue, and HashMap internals. All access is O(1) by index; no resizing, no generics for primitives.

Initialization 
// ── Declaration & allocation ──
int[] a = new int[5];                        // [0, 0, 0, 0, 0] — default-filled
int[] b = {1, 2, 3, 4, 5};                   // inline literal
int[] c = new int[]{1, 2, 3};                // explicit new + literal
String[] s = new String[3];                  // [null, null, null]

// ── Multi-dimensional ──
int[][] grid = new int[3][4];                // 3 rows × 4 cols, all 0
int[][] jagged = new int[3][];               // only rows allocated; cols later
jagged[0] = new int[]{1, 2};
jagged[1] = new int[]{3, 4, 5};

// ── Fill & copy ──
Arrays.fill(a, 7);                           // all elements → 7
int[] copy = Arrays.copyOf(b, b.length);     // shallow copy
int[] slice = Arrays.copyOfRange(b, 1, 4);   // [2, 3, 4]
int[] cloned = b.clone();                    // another way to copy

// ── char[] ↔ String ──
char[] chars = "hello".toCharArray();        // String → char[]
String str   = new String(chars);            // char[] → String
Key Points
Fixed size — length is set at creation and cannot change. Use ArrayList if you need dynamic sizing.
  • arr.length is a field, not a method (no parentheses).
  • Default values: 0 for numeric, false for boolean, null for objects.
No generics for primitives — int[] stores raw ints with zero boxing overhead. List<Integer> boxes every element.
  • For performance-critical code with primitives, prefer arrays over collections.
  • Arrays.asList(intArray) won’t work for primitives. Use IntStream.of(arr).boxed().toList() instead.
Pass by reference — arrays are objects. Passing an array to a method shares the same memory; mutations are visible to the caller.

char[] — works identically to int[] for all Arrays.* methods. See initialization card for String interop.

Arrays.sort(arr, (a, b) -> Integer.compare(a, b));

// ✅ Or use method reference

// Same issue with Comparator for int[] sorting

orig[0];  // 1 — unaffected ✅

📝 String, StringBuilder & Character

Strings are immutable in Java — every modification creates a new object. Use StringBuilder for mutable sequences (loops, concatenation). String is backed by a char[] (Java 8) or byte[] (Java 9+ compact strings).

Initialization 
// ── String ──
String s1 = "hello";                              // string pool (interned)
String s2 = new String("hello");                   // heap object, bypasses pool
String s3 = new String(charArray);                 // from char[]
String s4 = String.format("Hi %s, age %d", "Jo", 25); // formatted

// ── StringBuilder (mutable, not thread-safe) ──
StringBuilder sb = new StringBuilder();            // empty, default capacity 16
StringBuilder sb2 = new StringBuilder(100);        // initial capacity
StringBuilder sb3 = new StringBuilder("hello");    // from String
sb.append("world").append(42).insert(5, " ");      // chaining: "hello world42"
String result = sb.toString();                     // back to String
Key Points
Immutability — String objects cannot be changed after creation. Methods like substring, replace, toUpperCase all return a new String.
  • String concatenation in a loop creates n objects — use StringBuilder instead.
  • == compares references, not content. Always use .equals() for value comparison.
String pool — literals like "hello" are interned. "hello" == "hello" is true (same pool reference), but new String("hello") == "hello" is false.
  • str.intern() returns the pool reference for any String.
StringBuilder vs StringBuffer — same API, but StringBuffer is synchronized (slower). Always use StringBuilder unless sharing across threads.

🔍 Common Gotchas
Why do contains, indexOf, remove take Object not E?
  • They use equals(Object) internally — any two objects can be compared, so the parameter must accept Object.
  • A List<Number> may hold an Integer but you search with a Double — Object allows this cross-type lookup.
  • Generics are erased at runtime; restricting to E adds no real safety, only a compile-time illusion.
Collections only store boxed types — int → Integer, char → Character, double → Double. Java autoboxes automatically. 
// Pitfall: remove(int index) vs remove(Object value) on List<Integer>
List<Integer> list = new ArrayList<>(List.of(10, 20, 30));
list.remove(1);                    // removes element at INDEX 1 → [10, 30]
list.remove(Integer.valueOf(10));  // removes VALUE 10 → [30]

// Pitfall: different box types are never equal
new Integer(1).equals(new Long(1L));   // false!
Set<Long> set = Set.of(1L, 2L, 3L);
set.contains(1);                       // false — autoboxes to Integer, not Long
set.contains(1L);                      // true

// Pitfall: Integer cache (−128 to 127)
Integer a = 127, b = 127;
a == b;                                // true  — cached, same object
Integer x = 128, y = 128;
x == y;                                // false — different objects!
x.equals(y);                           // true  — always use .equals()
LinkedList: List + Queue + Deque — triple remove() overload
  • remove() (Queue) removes head
  • remove(Object) (Collection) removes by value
  • remove(int) (List) removes by index
Declare as the narrowest type you need (Queue or Deque) to avoid overload surprises.
// All three overloads visible on concrete LinkedList type
LinkedList<Integer> ll = new LinkedList<>(List.of(10, 20, 30));
ll.remove();                     // Queue.remove()   — removes head (10)
ll.remove(Integer.valueOf(30));  // Collection.remove(Object) — removes value 30
ll.remove(0);                    // List.remove(int)  — removes at index 0

// ✅ Narrow the type to limit which overloads are visible
Queue<Integer> q = new LinkedList<>();  // only Queue + Collection methods
Deque<Integer> d = new LinkedList<>();  // only Deque + Queue + Collection methods
TreeSet / TreeMap: equals() vs compareTo() disagreement
  • TreeSet/TreeMap use compareTo() (or the Comparator) for all operations: add, contains, remove, and dedup
  • If compareTo() returns 0 but equals() returns false, TreeSet treats them as duplicates even though they aren’t “equal”
  • HashSet uses equals() + hashCode() — same objects may behave differently in a TreeSet vs HashSet
// Case-insensitive comparator: "A" and "a" compare as 0
TreeSet<String> ts = new TreeSet<>(String.CASE_INSENSITIVE_ORDER);
ts.add("A");
ts.add("a");
ts.size();          // 1 — "a" treated as duplicate of "A" by compareTo
ts.contains("a");   // true — compareTo matches

// But equals() disagrees:
"A".equals("a");    // false!

// ⚠ Same objects in a HashSet — different result
HashSet<String> hs = new HashSet<>();
hs.add("A");
hs.add("a");
hs.size();          // 2 — "A" and "a" are NOT equal by equals()
